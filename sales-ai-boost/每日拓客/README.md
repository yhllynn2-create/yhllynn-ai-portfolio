# 客户商业价值研判与拓客引擎

> 一个项目干两件事——**① 拓客**：产出一批"大而近期有信号"、且**绝不撞上你在库存量客户**的新客户；**② 研判**：对任意已知公司评估经营价值，按 10 维度打分卡定级 S/A/B/C 并给资源建议。两者共用同一套"数据采集 + 7 节调研报告"引擎，打分卡是价值判定的统一标尺。

本子项目是一个 **WorkBuddy Skill**（`sales-daily-prospecting`）。它解决 B2B 销售两个高频痛点：**① 每天去哪找"值得跟"的新客户；② 如何评估手里的客户/目标"到底值多少资源"**——并且保证拓客不把已在跟的存量客户当新客重复报。

> 示例场景：某 B2B 厂商（示例：某头部硬件厂商）的深圳制造业销售，每天补 10 家"大而近期有信号"的新制造客户（排除已在库的两千多家），同时定期对手里存量客户做价值分层、把资源压到 S/A 级。本 Skill 把这套打法抽象成任何人可套用的流程——**方法开源，你的客户名单本地保管、绝不入库**。

## 一个洞察：拓客与研判本是同一件事

拓客看"这家新客有没有经营价值"，研判看"这家客户经营价值如何"——**用的是同一套评估维度**（成立时间、是否上市、人员规模、主营业务、盈利、工厂分布、行业地位、近期发展、未来趋势、近期招聘）。

所以本 Skill 让两者**共享引擎、统一标尺**：

```
        ┌──────────── 7 节调研报告（事实基底，共享）────────────┐
        │ 概况/工商/财务/招聘/融资/设备需求/切入建议            │
        └──────────────────────┬─────────────────────────────┘
                               │ 同一张打分卡（10 维度·权重·S/A/B/C）
              ┌────────────────┴────────────────┐
        拓客模式（信号发现+强排除）          研判模式（指定公司）
               ↓ 输出                               ↓ 输出
     "新客 + 价值层级"线索包            "经营价值结论 + 资源建议"
```

## 落地架构：中心化生产 + 全员分发

```
┌─────────────┐  产出线索包   ┌──────────────┐  复制跟进   ┌──────────┐
│  1 人跑引擎  │ ───────────▶ │ 汇总表/云文档  │ ──────────▶ │ 全体销售 │
│ (销售运营)   │  每个工作日   │  (统一分发)   │  零部署零学习 │ (只跟不写)│
└─────────────┘              └──────────────┘             └──────────┘
```

- 只需 **1 人** 按节奏触发，全体销售**零部署、零学习**，直接跟进。
- 强排除红线统一执行；价值分层统一标尺，避免"每人各评各的"。

## 核心能力

### 1. 信号驱动候选采集（拓客）
围绕"有资金 / 有扩编 = 有采购"逻辑，从招聘、融资、IPO、招投标、扩产新闻捞候选（详见 `references/信号源清单.md`）。

### 2. 强排除校验（拓客的护城河）
对候选与你团队的**在库/存量客户名单**做四重比对，零命中才过：归一化 / 双向子串 / `difflib` 模糊(≥0.85) / 别名母子公司穿透。算法与脚本见 `references/强排除校验方法.md` 与 `scripts/strong_exclusion.py`。

### 3. 7 节标准化调研报告（共享基底）
公司概况（含成立时间/人员）/ 工商与股权（含工厂产能）/ 财务表现 / 招聘信号 / 融资上市股价信号（含未来趋势）/ IT 设备与算力需求 / 切入建议。模板见 `references/调研报告模板.md`。

### 4. 商业价值打分卡（研判核心，拓客也用）
10 维度 × 权重（合计 100%）→ 加权总分 → 层级 S(≥4.0)/A(3.2)/B(2.4)/C(<2.4)；含红旗规则（任一维度=1 分不得直接判 S）。脚本 `scripts/value_score.py` 批量算。维度、权重、阈值见 `references/商业价值评估维度与打分卡.md`。

### 5. 价值研判报告（研判模式产出）
7 节基底 + ⑧ 商业价值打分卡 + ⑨ 研判结论与资源建议。模板见 `references/价值研判报告模板.md`。

## 工作流（两种模式）

**拓客模式**：定基准 → 采候选 → 强排除 → 深核验 → 写 7 节报告 → 跑打分卡定层级 + 交付（新客包）
**研判模式**：定基准（指定公司）→ （跳过强排除）→ 深核验 → 写 7 节报告 → 跑打分卡定层级 + 写 ⑧⑨ 结论 + 交付（研判报告）

## 5 分钟跑通（quickstart）

不想读文档也能立刻跑通？见 [`示例/quickstart.md`](./示例/quickstart.md) —— 用仓库自带的虚构演示数据，从「候选名单」一路跑到「docx 报告 + 汇总表」，零真实数据依赖。

**研判模式顺手试**：把示例报告里的 10 个维度分数写进 `scores.json`（`{"示例智能装备集团股份有限公司": {"经营稳定性":5, ...}}`），运行 `python scripts/value_score.py --json scores.json`，即得 `总分 4.45 层级：S`。

## 如何部署为 WorkBuddy Skill

```bash
# 把本目录整体复制到 WorkBuddy 用户级 skills 目录即可被识别
cp -r 每日拓客 ~/.workbuddy/skills/sales-daily-prospecting
```

部署后在对话中说「做今日拓客，找 10 家新客户」或「评估这家客户经营价值」即可触发；也可配置每个工作日 09:00 自动化。

## ⚠️ 机密与脱敏（非常重要）

- **你的在库/存量客户名单（含数千家公司名）属于商业机密，本仓库一律不收录**。脚本默认读取 `示例/incumbents_sample.json`（明显虚构的演示数据）。
- 真实名单请放在仓库**之外**的本地路径，运行 `strong_exclusion.py --list /你的/本地/名单.json` 时传入。
- 根目录与本目录的 `.gitignore`（或根 `.gitignore`）已加入排除规则，即使 `git add .` 也不会把真实名单误提交。
- 本 Skill 所有示例公司均为虚构，不含任何真实客户信息。打分结果（价值层级）同样本地保管，不与本仓库同库。

## 已知可优化点（待迭代）

- 别名表 `ALIAS` 需各团队按自身客户结构本地维护（不入库）。
- 信号采集目前依赖 WebSearch + 企查查/天眼查 MCP；可接入更多数据源（如招投标 API）。
- 汇总表列结构为通用 11 列（含价值层级）；如需对齐你部门既有大表，调整 `build_report.py` 的表头即可。
- 打分卡权重可按行业/产品线本地调参（保持合计 100%），存到本地即可，不与本仓库同库。
