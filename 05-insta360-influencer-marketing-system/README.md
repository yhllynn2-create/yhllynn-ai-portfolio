# 05 · 影石 Insta360 场景化达人投放决策系统

## 一、项目背景

Insta360 的产品价值高度依赖**真实使用场景**：骑行、滑雪、潜水、摩托、旅行记录……这些内容决定用户能否理解产品、产生兴趣并购买。达人因此同时承担「产品价值翻译」和「场景教育」两个角色，**选人质量直接影响内容表现、投放效率和新品进入市场的速度**。

但现有做法（平台搜索 + 达人库 + 人工经验，按粉丝量/互动率筛选）在面对**跨平台、跨场景的大规模候选池**时会失灵：

| 业务现象 | 直接影响 |
|---|---|
| 候选达人多、分布在多平台 | 人工覆盖有限，容易只盯已走红账号 |
| 通用指标无法描述内容质量 | 粉丝量相近，实际场景表达与转化能力差异巨大 |
| 推荐理由缺证据 | BD 难复核，也难向团队解释预算 |
| 历史合作没沉淀 | 每轮选人近乎从零开始，经验无法复用 |
| 大促/折扣/多渠道干扰 | 投后销量难归因到某位达人 |

## 二、解决方案

系统以一份**结构化任务简报**为起点（产品、目标市场、核心场景、预算、营销目标），在可接入的数据范围内生成候选池，完成硬筛、场景化评估与角色判断，最终产出**带证据的达人投资组合 + 预算区间 + 本地化合作 Brief**；投放结果回流历史库，用于规则校准与案例复用。

### 2.1 三层决策逻辑（核心设计）

把「筛选条件」和「核心判断」彻底拆开——这是整套方案的骨架。

| 决策层 | 回答的问题 | 处理方式 | 输出 |
|---|---|---|---|
| **资格层** | 能否进入本次任务 | 市场/语言/平台/受众地域/品牌安全/合规硬筛 | 合格候选池 |
| **匹配层** | 适合承担什么任务 | 结合卖点与场景，分析视频内容、受众互动、成长势能、历史表现 | 角色判断 + 适配分 + **证据卡** |
| **组合层** | 有限预算如何覆盖目标并分散风险 | 预算/场景覆盖/角色配比/集中度约束下构建组合 | 稳健型 & 进取型两套组合 |

### 2.2 业务主链（端到端）

```
任务简报 → 候选发现 → 资格硬筛 → 低成本初筛 → 场景深度评估
        → 角色判断与证据卡 → 达人投资组合 → 本地化合作 Brief
        → 投放与转化追踪 → 复盘与规则校准 ⟲（新案例回流历史库）
```

工程上采用**逐级收窄的分析漏斗**：大规模候选只跑低成本筛选，昂贵的视频多模态与评论语义分析只作用于少量深评对象——控制接口调用、算力成本与交付时间。每一层都保留「被排除原因」，可复查、可回溯。

### 2.3 四类专有能力（相对通用达人平台的差异）

1. **场景化匹配**——把产品卖点转成可观察的内容需求（第一视角稳定性、夜景运动、水下成像…），判断达人是否长期处在目标场景、能否自然承载产品、内容是否易被模仿扩散。
2. **推荐证据卡**——每个结论都绑定具体视频片段、评论样本、指标与数据来源+置信度；生成式模型只负责组织表达，**不允许补写数据中没有的事实**。
3. **达人投资组合**——按引爆/扩散/转化/潜力四种角色互补配置，而非单点粉丝量排名；报价用「区间 + 置信度」而非虚假精确单点。
4. **历史回放校准**——用严格时间切点复现历史合作，仅用「当时能拿到的数据」重跑引擎，与真实结果和传统基线（粉丝量/互动率）对照，持续校准权重阈值。

## 三、技术方案

深度绑定飞书生态作为交互与协同层，后端按可替换、可回放、成本分层三原则设计。

| 层 | 选型 | 作用 |
|---|---|---|
| 交互层 | 飞书 Aily 智能伙伴 / 妙搭应用 / 审批 / 消息卡片 | 自然语言建任务、证据卡与组合展示、预算放行、结果提醒 |
| 服务层 | FastAPI（Task / Creator / Portfolio / Brief / Review 五服务） | 按业务域拆分，统一以 `task_id` 为评分上下文 |
| 编排层 | Temporal（POC 阶段简化 Worker） | 候选发现、特征、媒体、评论、优化、生成的流水线编排 |
| 智能层 | LLM + RAG、VLM、ASR/OCR（FFmpeg）、Embedding、OR-Tools CP-SAT | 内容深评、评论语义、证据生成、组合优化 |
| 数据层 | PostgreSQL + pgvector、对象存储 | 结构化特征 + 向量检索 + 原始产物留存 |
| 可观测 | OpenTelemetry、MLflow | 日志/指标/Trace、模型与规则版本记录 |

> 设计原则：所有外部字段带 `observed_at`；历史验证只读 `decision_cutoff_at` 前的快照 → **保证时间可回放、杜绝时间穿越**。LLM/VLM/数据供应商均通过适配器调用，避免绑定单一服务。

## 四、验证方法论

方案不依赖「拍脑袋说必爆」，而是设计了一套**可证伪**的验证协议：

- **时间回放**：以合作开始日为切点，隐藏切点后数据，仅用当时信息重跑引擎，比较系统排序 vs 真实业务结果 vs 传统基线。
- **指标口径**：标签连续且质量好 → Spearman 秩相关 / NDCG；只能分高效/低效 → Precision@K / Recall / AUC。指标随标签结构选择，不硬套。
- **方法边界与适用性**：未合作达人只作未标注候选、不记为失败样本；历史样本存在选择偏差；预算/折扣/大促会干扰结果 → 回放检验的是**排序方向与规则校准**，而非严格因果证明。

这套「明确方法边界」的思路，是方案中工程化严谨性的体现——**不堆漂亮但站不住的结论，而是把判断建立在可验证的证据与可复现的流程上。**

| 能力维度 | 在本项目中的体现 |
